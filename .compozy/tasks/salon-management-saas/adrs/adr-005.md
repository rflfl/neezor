# ADR-005: Integração WhatsApp — Mock no MVP

## Status

Accepted

## Date

2025-05-23

## Context

O PRD define notificações via WhatsApp Business API como canal de comunicação com clientes (lembretes, confirmações). Decidiu-se mockar no MVP para não bloquear o desenvolvimento e evitar setup complexo (conta Business, chaves API, aprovação do Meta).

## Decision

Implementar **interface abstrata** `NotificationService` com driver `WhatsApp` + **mock driver** para MVP. Interface:

- `NotificationServiceInterface` define métodos: `sendReminder()`, `sendConfirmation()`, `sendCancellation()`, `sendPackageAlert()`
- Implementação MVP: `MockWhatsAppDriver` — loga no Laravel log (structured log) sem enviar nada
- Próxima fase: swap para `WhatsAppBusinessDriver` real (via Z-API, Twilio, ou SDK direto)

## Alternatives Considered

### Alternative 1: WhatsApp Business API própria com SDK

- **Description**: Integrar direto com WhatsApp Business API usando SDK PHP (como `netflie/laravel-whatsapp`)
- **Pros**: Sem intermediário, controle total
- **Cons**: Setup complexo (conta Business verificada, approval do Meta, rate limits, webhooks). Custo. Tempo de desenvolvimento.
- **Why rejected**: Tempo de setup não justifica no MVP. A lógica de negócio (quando enviar o quê) é mais importante que o canal.

### Alternative 2: Provedores (Z-API, Twilio, Eternity)

- **Description**: Usar provedor terceiro como Z-API, Twilio, ou Eternity para enviar mensagens
- **Pros**: API mais simples, menos setup. Alguns têm SDK Laravel.
- **Cons**: Custo mensal, dependência de terceiros, configuração de credenciais por salão. Complexidade extra no MVP.
- **Why rejected**: Custo operacional adicional desde o MVP. Setup ainda significativo. Melhor mockar e integrar depois com mais contexto.

### Alternative 3: Integração via webhook para serviço externo

- **Description**: Webhook para bot de WhatsApp existente do salão
- **Pros**: Salão usa ferramenta que já conhece
- **Cons**: Incompatibilidade entre bots, sem controle, debugging difícil
- **Why rejected**: Dependência de terceiros sem controle sobre formato de mensagem.

## Consequences

### Positive

- Desenvolvimento não bloqueado por credenciais de WhatsApp
- Interface limpa (`NotificationServiceInterface`) que pode ser implementada com qualquer driver
- Lógica de quando e o que enviar fica pronta antes do canal real
- Testes fáceis: mock driver permite assertions sem enviar mensagens

### Negative

- MVP não envía mensagens reais
- Feature de WhatsApp só funciona depois de integrar driver real
- Risco de não testar fluxo real de mensagens antes do lançamento

### Risks

- **Risco**: Interface muda quando driver real é integrado
  - **Mitigação**: Definir interface completa agora (todos os métodos) para minimizar mudanças futuras
- **Risco**: Mock driver esconde bugs de formatação de mensagem
  - **Mitigação**: Testar output do mock driver (log assertions) para garantir formato correto

## Implementation Notes

- Interface em `Domain/Notifications/Contracts/NotificationServiceInterface.php`
- Mock em `Domain/Notifications/Drivers/MockWhatsAppDriver.php`
- Registro em Service Provider: `app()->bind(NotificationServiceInterface::class, MockWhatsAppDriver::class)`
- Structured log em todas as chamadas do mock driver:
  ```php
  Log::info('WhatsApp notification', [
      'type' => 'reminder',
      'client_phone' => '+55...',
      'message' => '...',
      'scheduled_at' => '...',
  ]);
  ```
- Para próxima fase: criar `WhatsAppBusinessDriver` implementar interface
