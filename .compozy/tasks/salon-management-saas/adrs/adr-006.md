# ADR-006: Framework de Testes — PHPUnit

## Status

Accepted

## Date

2025-05-23

## Context

O PRD define que testes são obrigatórios para regras de negócio financeiras (comissão, caixa, DRE). PHPUnit é o framework padrão do Laravel. Pest é um wrapper mais moderno. Jest/Vitest seria para testes de componentes Vue frontend.

## Decision

Usar **PHPUnit 12+** (já instalado no scaffold) para todos os testes:

- **Unit tests**: Domain services (CommissionService, CashboxService, AvailabilityService, etc.)
- **Feature tests**: Rotas, controllers, fluxos completos
- **Database: SQLite in-memory** para testes rápidos (sem MySQL dependency nos tests)

## Alternatives Considered

### Alternative 1: Pest

- **Description**: Wrapper sobre PHPUnit com syntax mais limpa (closures, describe/it)
- **Pros**: Syntax mais legível, less boilerplate. Suporte a PHPUnit nativo.
- **Cons**: Dependência adicional, curva de aprendizado pequena, manfaat não justifica o overhead para equipe pequena.
- **Why rejected**: PHPUnit é suficiente para MVP. Syntax PHPUnit é bem estabelecida. A diferença de produtividade é marginal.

### Alternative 2: PHPUnit + Jest/Vitest (duplo framework)

- **Description**: PHPUnit para backend, Jest/Vitest para componentes Vue frontend
- **Pros**: Cobertura completa (backend + frontend)
- **Cons**: Duas configurações separadas, duas curvas de aprendizado, mais overhead de manutenção. Componentes Vue no MVP são simples (Inertia pages com props).
- **Why rejected**: Custo de manter dois frameworks não se justifica no MVP. PHPUnit pode testar fluxos que incluem Vue components via Inertia feature tests.

## Consequences

### Positive

- PHPUnit já instalado e configurado (scaffold Jetstream)
- Convenções Laravel bem estabelecidas (tests/TestCase.php, tests/Feature/, tests/Unit/)
- Suporte nativo Laravel (RefreshDatabase, WithoutMiddleware, etc.)
- SQLite in-memory para testes extremamente rápidos
- Cobertura obrigatória de regras de negócio financeiras (comissão, caixa, DRE)

### Negative

- Syntax um pouco mais verbosa que Pest (não afeta produtividade significativamente)
- Testes de componentes Vue isolados não são possívels (requer Jest/Vitest)

### Risks

- **Risco**: Testes lentos com banco de dados real
  - **Mitigação**: Usar `RefreshDatabase` com trait SQLite in-memory. Factory seeding leve.
- **Risco**: Cobertura insuficiente de regras de domínio
  - **Mitigação**: Regra explícita — todo service em `Domain/*` deve ter testes unitários cobrindo 100% dos métodos públicos.

## Implementation Notes

- Estrutura de testes:
  ```
  tests/
    Unit/
      Domain/
        Commission/
        Cashbox/
        Scheduling/
        Notifications/
        Packages/
    Feature/
      Http/
        Controllers/
      Pages/
  ```
- Database em testes: `testing` connection com `sqlite` em memória
- Trait `PerformsTenantTests` para setup de tenant em cada test
- Factories para todos os models principais
- Coverage mínimo: 80% overall, 100% para domínio financeiro
