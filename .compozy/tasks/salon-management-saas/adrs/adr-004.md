# ADR-004: Sistema de Agenda — Custom Scheduler

## Status

Accepted

## Date

2025-05-23

## Context

O PRD define agenda com slots de disponibilidade, conflito de horários, e auto-agendamento web. A pergunta era: implementar do zero ou usar package existente?

## Decision

Implementar **sistema de agenda custom** (Domain/Scheduling) do zero, com:

- **Modelagem de slots**: Horários disponíveis gerados a partir de configuração de agenda por profissional e por serviço
- **Motor de disponibilidade**: Calcula slots livres baseado em agenda existente + buffer entre atendimentos
- **Conflito detection**: Valida no momento do agendamento se há sobreposição
- **Auto-agendamento via API pública**: Endpoint Inertia para portal do cliente

## Alternatives Considered

### Alternative 1: Pacote Laravel existente (spatie/laravel-schedule ou similar)

- **Description**: Usar package de terceiros para gerenciar schedule/availability
- **Pros**: Menos código custom
- **Cons**: Packages de schedule focam em cron/job scheduling, não em disponibilidade de salão. Poucos packages bons para este caso específico. Adiciona dependência externa.
- **Why rejected**: Packages existentes são para server scheduling (cron-like), não para disponibilidade de serviços/profissionais. O domínio de salão é específico demais para um package genérico fazer bem.

### Alternative 2: Laravel native scheduler (app/Console/Kernel)

- **Description**: Usar Laravel native cron/scheduler para jobs recorrentes
- **Pros**: Laravel-native, tested, reliable
- **Cons**: Laravel scheduler é para jobs recorrentes do servidor, não para disponibilidade de agenda. Não resolve a lógica de slots e conflitos.
- **Why rejected**: Resolve um problema diferente. Laravel scheduler é bom para reminders, não para gestão de disponibilidade.

## Consequences

### Positive

- Controle total sobre a lógica de disponibilidade (específico do domínio de salão)
- Modelo de dados feito sob medida (profissionais, serviços, durações, buffers)
- Flexibilidade para iterar rápido em cima de regras de negócio
- Sem dependência externa

### Negative

- Mais código para escrever e manter
- Bugs em lógica de disponibilidade podem causar double-booking (crítico)
- Necessidade de testes rigorosos para cobertura de edge cases (feriados, buffers, etc.)

### Risks

- **Risco**: Double-booking se lógica de conflito tiver bug
  - **Mitigação**: Testes unitários cobrindo todos os cenários de conflito + constraint de banco (unique index) como última linha de defesa
- **Risco**: Lógica de slots complexa demais (duração variável por serviço, buffers diferentes, etc.)
  - **Mitigação**: MVP prioriza serviços com duração fixa, buffers mínimos. Complexidade incremental nas próximas fases.

## Implementation Notes

- Entidade `ScheduleSlot` (ou similar) representa um bloco de disponibilidade
- Service `AvailabilityService` calcula slots livres
- Validação de conflito no Repository/Service antes de persistir
- Índice único composto em `(tenant_id, professional_id, start_time, end_time)` como constraint de banco
- Endpoints públicos via rota `/booking/{tenant}` autenticada com token público (UUID link)
