# ADR-002: Estratégia de Multi-Tenancy — tenant_id com Global Scopes

## Status

Accepted

## Date

2025-05-23

## Context

O Neezor é multi-tenant: cada salão (tenant) deve ver apenas seus próprios dados. Decidiu-se usar uma abordagem por `tenant_id` em todas as tabelas, mas era necessário definir COMO esse filtro seria aplicado de forma consistente e sem erros.

As opções eram:
- **Global Scopes do Eloquent** — aplica filtro automático em todas as queries do model
- **Package stancl/tenancy** — biblioteca completa com tenancy features extras
- **Custom Trait Tenantable** — trait com observer para aplicar tenant_id

## Decision

Usar **Global Scopes do Eloquent** via trait `BelongsToTenant` para aplicar automaticamente `tenant_id` em todas as queries dos models. Implementação:

1. Trait `BelongsToTenant` adiciona `tenant_id` automaticamente em `creating` events
2. Trait `ScopeTenantAware` aplica global scope que filtra `tenant_id` em todas as queries
3. Middleware `EnsureTenantIsSet` garante que `tenant_id` está configurado no request atual
4. Trait aplicada a TODOS os models de domínio

## Alternatives Considered

### Alternative 1: Package stancl/tenancy

- **Description**: Biblioteca completa com tenancy completo (migrations separadas, tenant identification automático, etc.)
- **Pros**: Funcionalidades extras prontas (tenant routing, job tenancy, etc.)
- **Cons**: Overhead desnecessário para MVP. Complexity extra para algo simples. Dependência externa com curva de aprendizado.
- **Why rejected**: O MVP não precisa de features avançadas de tenancy. A complexidade do package não se justifica para 1 salão por conta.

### Alternative 2: Custom Trait Tenantable

- **Description**: Trait com Observer que aplica tenant_id em cada model manualmente
- **Pros**: Total controle, sem magic
- **Cons**: Requer discipline manual em cada model (é fácil esquecer de aplicar). Não escala bem.
- **Why rejected**: Risco de vazamento de dados se um developer esquecer de aplicar o trait. Global Scope é mais robusto.

## Consequences

### Positive

- Filtro automático em todas as queries — impossível esquecer de filtrar por tenant
- Implementação simples e Laravel-native
- Easy to understand: cada model sabe que pertence a um tenant
- Boa performance: index em `tenant_id` + where clause implícito

### Negative

- Queries raw (DB::select) precisam de filtro manual
- Testes precisam mockar tenant_id em cada test
- Pode gerar confusão em queries complexas (joins multi-tenant)

### Risks

- **Risco**: Queries raw escapam do filtro de tenant
  - **Mitigação**: Criar base de testes com cobertura de multi-tenancy, audit de código, documentação
- **Risco**: Developer esquece de adicionar trait em novo model
  - **Mitigação**: Template de model com trait já inclusa, code review obrigatório, PHPStan rule

## Implementation Notes

- Criar trait `BelongsToTenant` em `App/Models/Traits/BelongsToTenant.php`
- Criar trait `ScopeTenantAware` em `App/Models/Traits/ScopeTenantAware.php`
- Criar global scope em `App/Models/Scopes/TenantScope.php`
- Middleware em `App/Http/Middleware/EnsureTenantIsSet.php`
- Todos os models devem usar ambas as traits
- Testes: trait de teste `PerformsTenantTests` para facilitar setup
