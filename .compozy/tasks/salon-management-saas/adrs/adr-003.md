# ADR-003: Banco de Dados — MySQL

## Status

Accepted

## Date

2025-05-23

## Context

O PRD define que o banco pode ser PostgreSQL ou MySQL. Ambos são suportados pelo Laravel. A decisão técnica precisa ser tomada antes de criar migrations e models do MVP. O servidor de produção ainda não foi definido, então a escolha deve considerar popularidade no mercado brasileiro e suporte do ecossistema Laravel.

## Decision

Usar **MySQL 8.0+** como banco de dados para o Neezor MVP.

## Alternatives Considered

### Alternative 1: PostgreSQL

- **Description**: Banco relacional open-source com suporte a JSON, full-text search, e tipos avançados
- **Pros**: Tipos nativos mais ricos (JSONB, arrays, tsvector). Suporte multi-tenant com schemas é nativo. Melhor para analytics.
- **Cons**: Historicamente menos popular na comunidade brasileira de hosting compartilhado. Algumas ferramentas Laravel têm suporte melhor para MySQL. SQLite é development-only (não escala para produção).
- **Why rejected**: MySQL é mais acessível para hospedagem brasileira (hostings compartilhados, VPS comuns). A complexidade do PostgreSQL não se justifica para o MVP. Laravel suporta ambos nativamente.

### Alternative 2: SQLite

- **Description**: Banco em arquivo único, sem servidor, ideal para desenvolvimento
- **Pros**: Zero configuração. Rápido para development. Excelente para testes.
- **Cons**: Não serve para produção multi-tenant com múltiplos concurrent users. Limitações em concurrent writes, locks, tipos de dados.
- **Why rejected**: O PRD define MySQL/PostgreSQL para produção. SQLite não escala para produção real.

## Consequences

### Positive

- Amplo suporte de hosting no Brasil (hostings compartilhados, VPS, managed services)
- Laravel-native com performance otimizada
- Ferramentas de administração maduras (phpMyAdmin, MySQL Workbench, DBeaver)
- Suporte a múltiplas conexões para migrations/testes
- Suporte a storage engines (InnoDB para produção)

### Negative

- Migration de PostgreSQL → MySQL pode ter incompatibilidades de sintaxe
- Alguns tipos avançados do PostgreSQL não estão disponíveis

### Risks

- **Risco**: MySQL 5.x em vez de 8.x (versão antiga com limitações)
  - **Mitigação**: Especificar MySQL 8.0+ na documentação de deploy, checar versão no setup
- **Risco**: Encoding e collation diferentes entre ambientes
  - **Mitigação**: Definir `charset: utf8mb4` e `collation: utf8mb4_unicode_ci` em config/database.php

## Implementation Notes

- Config em `config/database.php`: `'default' => env('DB_CONNECTION', 'mysql')`
- `.env.example`: `DB_CONNECTION=mysql`, `DB_HOST=127.0.0.1`, `DB_PORT=3306`, `DB_DATABASE=neezor`, `DB_USERNAME=root`, `DB_PASSWORD=`
- Config de migration em `config/database.php`:
  ```php
  'mysql' => [
      'driver' => 'mysql',
      'url' => env('DATABASE_URL'),
      'host' => env('DB_HOST', '127.0.0.1'),
      'port' => env('DB_PORT', '3306'),
      'database' => env('DB_DATABASE', 'neezor'),
      'username' => env('DB_USERNAME', 'root'),
      'password' => env('DB_PASSWORD', ''),
      'unix_socket' => env('DB_SOCKET', ''),
      'charset' => 'utf8mb4',
      'collation' => 'utf8mb4_unicode_ci',
      'prefix' => '',
      'prefix_indexes' => true,
      'strict' => true,
      'engine' => 'InnoDB',
      'options' => extension_loaded('pdo_mysql') ? array_filter([
          PDO::MYSQL_ATTR_SSL_CA => env('MYSQL_ATTR_SSL_CA'),
      ]) : [],
  ],
  ```
- Usar SQLite para testes e development local (config já existe no scaffold)
