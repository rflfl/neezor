# Rules – Neezor (Salon Management SaaS)Estas regras são **sempre ativas** para qualquer pessoa ou agente que trabalhe neste repositório.Não ignore ou sobrescreva estas regras sem alinhamento explícito.---## 1. Arquitetura por domínio1.1 Estrutura geral  - A aplicação Neezor é organizada **por domínios de negócio**, não por camadas técnicas genéricas.  - Cada domínio de negócio deve possuir seu próprio módulo, por exemplo:  - `Domain/Scheduling`   – agenda e horários  - `Domain/Cashbox`      – caixa diário, lançamentos, fechamento  - `Domain/Commission`   – regras de comissão e rateio  - `Domain/Customers`    – clientes  - `Domain/Staff`        – profissionais/prestadores de serviço  - `Domain/Services`     – serviços/procedimentos oferecidos1.2 Local da regra de negócio  - **Toda regra de negócio** deve viver dentro do domínio correspondente, e não em controllers,  jobs, observers ou helpers soltos.  - Controllers, commands e outras camadas de interface são apenas **orquestradores**  que delegam para serviços/casos de uso do domínio.1.3 Criação de novas funcionalidades  - Ao implementar uma nova feature, identifique o domínio de negócio correspondente.  - Se não existir um domínio adequado:  - Proponha a criação de um novo domínio.  - Atualize `docs/domain-salao.md` descrevendo o novo domínio e suas responsabilidades.- É **proibido** criar pastas genéricas como `App/Services` sem vínculo explícito com um domínio.---## 2. Multi-tenancy e isolamento de dados2.1 Isolamento por salão  - Neezor é multi-tenant: **toda operação** de leitura ou escrita deve ser filtrada  pelo tenant atual (salão/organização).  - É proibido acessar ou retornar dados de outro tenant.2.2 Regras práticas  - Queries devem sempre incluir o filtro de tenant (ex.: `where('tenant_id', $tenantId)` ou equivalente).  - Nunca derive dados de tenant por suposição implícita (ex.: não usar apenas subdomínio/host sem confirmação).  - Em joins entre tabelas, garanta que **todas** respeitam o mesmo `tenant_id`.2.3 Testes de multi-tenancy  - Sempre que possível, crie testes que:  - Confirmem que dados de um tenant não são visíveis em outro.  - Validem que seeds/tests não vazam dados entre tenants.---## 3. Estilo de código e padrões3.1 Linguagem e stack  - Linguagem padrão: **PHP 8.x**.  - Framework: **Laravel**.  - Padrão de frontend: Vue 3 + Inertia.js, UI com shadcn-vue + Tailwind.3.2 Convenções  - Siga convenções PSR e padrões já estabelecidos no código existente.  - Nomenclatura:  - Classes: `PascalCase`  - Métodos/variáveis: `camelCase`  - Constantes: `UPPER_SNAKE_CASE`- Mantenha funções e métodos pequenos e focados em uma responsabilidade.3.3 Dependências  - Evite adicionar dependências externas sem justificativa clara.  - Para lógica de domínio, prefira criar casos de uso/serviços no domínio  antes de buscar bibliotecas de terceiros.---## 4. Regras de negócio financeiras4.1 Comissões e parte do salão  - Qualquer cálculo de comissão deve seguir as regras descritas em `docs/sdd/*`  para o domínio de comissões/caixa.  - É proibido alterar a fórmula de cálculo de comissão sem:  - Atualizar a documentação de domínio e a spec SDD correspondente.    - Adicionar ou ajustar testes cobrindo o novo comportamento.4.2 Caixa e lançamentos  - Abertura, lançamentos e fechamento de caixa devem manter **consistência contábil**:  - Nenhum lançamento pode ficar sem referência a atendimento, profissional ou origem.    - Ajustes manuais devem ser rastreáveis (com motivo e responsável).4.3 Lucro do mês  - O cálculo de lucro do mês deve ser determinístico e baseado em:  - Receitas (serviços, vendas) – por tenant e por período.    - Despesas/custos (produtos, taxas, etc.) – por tenant e por período.  - Qualquer alteração nessa regra exige testes de regressão com cenários claros  (ex.: mês com vários atendimentos, sem atendimentos, com descontos, etc.).---## 5. Logging e observabilidade5.1 O que logar  - Logue eventos relevantes de domínio (ex.: fechamento de caixa, alteração de comissão,  cancelamento de agendamento).  - Registre erros e exceções com contexto suficiente para diagnóstico  (IDs, tipos de entidade, status).5.2 O que **não** logar  - É proibido logar:  - Senhas, tokens, dados de cartão, credenciais de acesso.    - Dados pessoais sensíveis completos (ex.: documento completo, dados bancários).5.3 Ferramentas de log  - Use a infraestrutura de logs padrão do framework (ex.: `Log::info`, `Log::error`)  ou abstrações já presentes no projeto.  - Não use `var_dump`, `dd` ou `print_r` em código de produção.---## 6. Testes e qualidade6.1 Obrigatoriedade  - Toda nova regra de negócio deve ser coberta por testes (unitários ou de integração).  - Correções de bug devem incluir um teste que falha antes e passa depois da correção.6.2 Comandos padrão  - Antes de considerar uma tarefa concluída, execute:  - `php artisan test` ou `./vendor/bin/pest`    - `phpstan analyse` (se configurado)    - `npm run lint` e, quando fizer sentido, `npm run build -- --dry-run` para frontend6.3 Testes estáveis  - Evite testes flakey (dependentes de horário, ordem de execução, serviços externos).  - Use factories e seeds para garantir cenários reprodutíveis.---## 7. Migrations e dados7.1 Migrations  - Toda alteração estrutural de banco deve passar por migration versionada.  - Não altere diretamente o schema no banco sem migration correspondente.7.2 Compatibilidade  - Ao alterar colunas usadas em produção (como valores financeiros ou campos de relacionamento):  - Mantenha compatibilidade com dados existentes ou escreva scripts claros de migração.    - Adicione testes que validem o comportamento com dados pré-existentes.7.3 Seeds  - Seeds de desenvolvimento não devem poluir ou comprometer dados de produção.  - Seeds de teste devem ser determinísticos e documentados quando usados em cenários críticos.---## 8. Segurança, autenticação e privacidade8.1 Autenticação e autorização (Jetstream + Inertia)  - A autenticação base deve continuar sendo fornecida pelo Laravel Jetstream  usando a stack Inertia.  - Customizações em fluxos de login/registro/profile devem respeitar:  - Rotas e middlewares já definidos pelo Jetstream.    - Proteções de verificação de email, 2FA e gerenciamento de sessão quando habilitados.- Nunca desabilite checks de autenticação/autorização em código de produção.8.2 Dados de clientes e profissionais  - Trate dados pessoais com cuidado e, quando possível, minimize exposição.  - Evite retornar mais dados do que o necessário em respostas de API e páginas Inertia.8.3 Exceções  - Mensagens de erro não devem revelar detalhes internos (queries, paths sensíveis,  stack traces completos).---## 9. Git, branches e PRs9.1 Commits  - Commits devem ser pequenos, coesos e com mensagens descritivas.  - Evite commits de “refactor gigante” misturado com mudanças de regra de negócio.9.2 Pull Requests  - PRs devem:  - Descrever claramente o problema e a solução.    - Mencionar o domínio afetado (ex.: `[Domain: Cashbox]`).    - Incluir evidências de testes executados (comandos e resultados).9.3 Código morto  - É proibido manter código morto/commentado “para uso futuro”.  - Sempre remova código que não é mais necessário.---## 10. Frontend (Vue + Inertia + shadcn-vue)10.1 Organização  - Páginas devem ser implementadas como Inertia pages em `resources/js/Pages/**`.  - Componentes reutilizáveis devem ficar em `resources/js/Components/**`,  seguindo o padrão do Jetstream + shadcn-vue.10.2 Sistema de design (`DESIGN.md`)  - Para qualquer tarefa de UI/UX:  - Consulte `DESIGN.md` na raiz do projeto para tokens de design, componentes padrão    e exemplos de layout.    - Reutilize componentes de UI existentes antes de criar novos.- Novos componentes visuais devem:  - Seguir o sistema de design definido em `DESIGN.md`.    - Reutilizar tokens de Tailwind e padrões de espaçamento já existentes.10.3 Limites  - É proibido introduzir outro framework de frontend (React, Alpine, etc.) para novas telas.  - Evite lógica de negócio em componentes Vue; mantenha a regra de negócio nos domínios.---## 11. Ações proibidasEstas ações são **sempre proibidas**, salvo instrução explícita em contrário:- Criar lógica de negócio nova fora dos domínios (`Domain/*`).  - Alterar ou remover checks de multi-tenancy.  - Desativar testes ou verificações de qualidade apenas para “passar o build”.  - Expor dados de um tenant para outro, mesmo em ambientes de desenvolvimento.  - Alterar arquivos de infraestrutura crítica (CI/CD, deploy, configuração sensível)  sem que a tarefa seja explicitamente sobre isso.  - Comentar ou apagar testes existentes sem justificativa clara e registro no PR.  - Logar dados sensíveis (senhas, tokens, dados de cartão, documentos completos).  - Introduzir stack de frontend adicional paralela a Vue + Inertia + shadcn-vue.---Ao trabalhar no Neezor, trate estas regras como **contrato** entre o time, os agentes de IA e o produto.Qualquer desvio planejado deve ser documentado, justificado e, idealmente, temporário.
